# Power Automate Desktop (PAD) and Queue Processing

# 1. What is Power Automate Desktop?

Power Automate Desktop (PAD) is Microsoft's RPA tool used to automate desktop and legacy applications by simulating human actions such as:

* Mouse clicks
* Keyboard actions
* File operations
* Excel automation
* Web automation
* SAP/Desktop application automation

Interview Answer:

Power Automate Desktop is an RPA tool that automates repetitive tasks on desktop and web applications using UI and API automation.

---

# 2. Types of Desktop Flows

## Attended Desktop Flow

* Triggered by users.
* Runs on user's machine.
* User interaction may be required.

Example:
User clicks a button in Power Apps → Desktop Flow starts.

---

## Unattended Desktop Flow

* Runs without user interaction.
* Executes on dedicated machines.
* Triggered from Cloud Flows.

Example:
Every night at 10 PM a desktop automation runs automatically.

---

# 3. How Cloud Flow Connects to Desktop Flow

Typical Architecture:

Power Apps/User
↓
Cloud Flow
↓
Run Desktop Flow Action
↓
Machine / Machine Group
↓
Desktop Flow

Cloud Flow acts as an orchestrator.

---

# 4. Does PAD Have Queues Like UiPath?

No.

Power Automate Desktop does not provide native Queue Items like UiPath Orchestrator.

To implement queue processing, we generally use:

* Dataverse Table
* SharePoint List
* SQL Table
* Excel Table

These act as custom queues.

---

# 5. Using SharePoint as Queue

Example:

SharePoint List: InvoiceQueue

| ID | InvoiceNo | Status |
| -- | --------- | ------ |
| 1  | INV001    | New    |
| 2  | INV002    | New    |
| 3  | INV003    | New    |

Status Values:

* New
* In Progress
* Completed
* Failed

# Flow Process:

Cloud Flow
↓
Get Items where Status='New'
↓
Update Status='In Progress'
↓
Run Desktop Flow
↓
Desktop processes invoice
↓
Update Status='Completed'

Interview Question:

# Q: How did you implement queue processing in Power Platform?

Answer:

Since Power Automate Desktop does not provide native queues, we implemented a custom queue using SharePoint Lists/Dataverse. Cloud Flows retrieved pending records, invoked Desktop Flows for processing, and updated record statuses after completion.

---

# 6. Using Dataverse as Queue (Recommended)

Create Table:

ProcessingQueue

Columns:

* RecordID
* CustomerID
* Status
* RetryCount
* ErrorMessage
* CreatedDate

Example:

| RecordID | Customer | Status |
| -------- | -------- | ------ |
| 1001     | Vodafone | New    |
| 1002     | Airtel   | New    |

Processing Steps:

Step 1:

Cloud Flow triggers.

Step 2:

List rows where:

```text
Status = 'New'
```

Step 3:

Update row:

```text
Status = 'In Progress'
```

Step 4:

Call Desktop Flow.

Step 5:

Desktop Flow returns result.

Step 6:

Update row:

```text
Status = 'Completed'
```

or

```text
Status = 'Failed'
```

# Scheduling and Triggering Power Automate Desktop Flows

# 1. Can We Schedule a Desktop Flow Directly?

**No.**

Power Automate Desktop (PAD) flows cannot be scheduled directly from the desktop application.

Desktop Flows are usually:

* Triggered by Cloud Flows
* Triggered manually by users
* Triggered through Event-based Cloud Flows

The Cloud Flow acts as the orchestrator.

---

# 2. How are Desktop Flows Triggered?

There are mainly three ways:

## A. Manual Trigger (Attended Automation)

User manually starts the Desktop Flow.

Example:

User opens Power Automate Desktop and clicks **Run**.

OR

User clicks a button in a Canvas App.

Architecture:

User
↓
Cloud Flow
↓
Run Desktop Flow

Interview Question:

Q: How do users manually trigger Desktop Flows?

Answer:

Users can manually run Desktop Flows directly from Power Automate Desktop or invoke them through Instant Cloud Flows.

---

## B. Scheduled Trigger (Most Common)

A Cloud Flow with a **Recurrence Trigger** schedules the Desktop Flow.

Example:

Every day at 10 PM

↓

Cloud Flow starts

↓

Run Desktop Flow action executes

Architecture:

Recurrence Trigger
↓
Cloud Flow
↓
Run Desktop Flow
↓
Machine/Machine Group
↓
Desktop Flow

Example Scenario:

Daily invoice processing at midnight.

Interview Question:

Q: How do you schedule Desktop Flows?

Answer:

Desktop Flows are scheduled using Cloud Flows with a Recurrence trigger. The Cloud Flow invokes the Desktop Flow on a configured machine or machine group.

---

# 3. Event-Based Triggering

Desktop Flows can start based on business events.

Examples:

## SharePoint Trigger

When an item is created

↓

Run Desktop Flow

## Email Trigger

When an email arrives

↓

Run Desktop Flow

## Dataverse Trigger

When a row is added

↓

Run Desktop Flow

Example:

Customer request submitted in Power Apps

↓

Record saved in Dataverse

↓

Cloud Flow triggers

↓

Desktop Flow processes request in a legacy application

---

# 4. "Run a Desktop Flow" Action

Cloud Flows use the action:

```text
Run a desktop flow
```

Configuration requires:

* Desktop Flow
* Machine or Machine Group
* Credentials (for unattended)
* Input Parameters

---

# 5. Machine and Machine Group

## Machine

Represents a single computer where Desktop Flows execute.

Example:

VM01

---

## Machine Group

Collection of multiple machines.

Benefits:

* Load balancing
* High availability
* Scalability

Example:

Machine Group: FinanceBots

Contains:

* VM01
* VM02
* VM03

Interview Question:

Q: Why do we use Machine Groups?

Answer:

Machine Groups improve scalability and distribute Desktop Flow execution across multiple machines.

---

# 6. Attended vs Unattended Triggering

| Attended         | Unattended                      |
| ---------------- | ------------------------------- |
| User starts flow | Cloud Flow starts automatically |
| User logged in   | No user interaction             |
| Desktop visible  | Runs in background              |
| Manual execution | Scheduled execution             |

---

# 7. Prerequisites for Unattended Execution

Before scheduling:

1. Install Power Automate Machine Runtime.
2. Register machine in Power Platform.
3. Create Machine or Machine Group.
4. Configure machine credentials.
5. Assign unattended license.
6. Grant required permissions.

---

# 8. Real-Time Example

Requirement:

Every night at 11 PM process customer data in a legacy application.

Implementation:

Step 1:

Create Scheduled Cloud Flow.

Trigger:

```text
Recurrence
Frequency = Daily
Time = 11 PM
```

Step 2:

Add action:

```text
Get rows from Dataverse
```

Step 3:

Loop through records.

Step 4:

Use:

```text
Run a desktop flow
```

Step 5:

Desktop Flow opens legacy application and processes records.

Step 6:

Update Dataverse status.

---

# 9. Common Interview Questions

## Q: Can Desktop Flows be triggered directly?

Answer:

Desktop Flows are typically triggered manually or invoked through Cloud Flows.

---

## Q: How do you schedule Desktop Flows?

Answer:

Desktop Flows are scheduled using Scheduled Cloud Flows with a Recurrence trigger.

---

## Q: How does a Cloud Flow invoke a Desktop Flow?

Answer:

Cloud Flows use the "Run a desktop flow" action to invoke Desktop Flows on configured machines or machine groups.

---

## Q: What triggers have you used to start Desktop Flows?

Answer:

I have used Recurrence triggers, SharePoint triggers, Dataverse triggers, email triggers, and Power Apps triggers to invoke Desktop Flows.

---

"Power Automate Desktop Flows cannot be scheduled directly. We use Cloud Flows with triggers such as Recurrence, SharePoint, Dataverse, or Power Apps, and then invoke the Desktop Flow using the 'Run a desktop flow' action on a configured machine or machine group."

# 6. Selectors in PAD (Very Important)

Selectors uniquely identify UI elements.

Types:

## UI Element Selectors

Used for:

* Buttons
* Text fields
* Menus

Good selectors should:

* Be stable
* Avoid dynamic attributes
* Use reliable properties

Interview Question:

Q: What challenges did you face in UI automation?

Answer:

The main challenges were dynamic UI elements, application latency, and unstable selectors. I resolved these issues using proper waits, reliable selectors, retry mechanisms, and image/keyboard automation where necessary.

# 8. Difference Between Web, UI and Desktop Automation

| Feature                | Web Automation | UI Automation        | Desktop Automation      |
| ---------------------- | -------------- | -------------------- | ----------------------- |
| Target                 | Browser Apps   | Any UI Element       | Windows Applications    |
| Example                | Chrome, Edge   | Buttons, Menus       | SAP, Legacy Apps        |
| Uses Browser Extension | Yes            | No                   | No                      |
| Speed                  | Fast           | Medium               | Medium                  |
| Reliability            | High           | Depends on Selectors | Depends on UI Stability |
